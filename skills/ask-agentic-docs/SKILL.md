# Ask Agentic Documentation

**Trigger**: `/ask`  
**Purpose**: Query agentic documentation through an in-memory NetworkX knowledge graph

## Overview

This skill provides controlled access to agentic documentation by loading the knowledge graph into memory and performing graph-based retrieval. All data is retrieved exclusively through the in-memory NetworkX graph.

## Prerequisites

**Required**: The repository must contain a knowledge graph at:
```
<repo>/agentic/knowledge-graph/graph.json
```

**Format**: NetworkX node-link JSON format (generated by `/create` skill)

## Validation

Before executing any query, the skill validates:

1. ✅ **Graph exists**: `<repo>/agentic/knowledge-graph/graph.json` file present
2. ✅ **Valid format**: JSON is valid NetworkX node-link format
3. ✅ **Not empty**: Graph contains nodes and edges

**If validation fails**: Return error message with instructions to run `/create` first.

## Workflow

### Step 1: Validate Graph Exists

```python
import os
from pathlib import Path

# Check if graph exists
graph_path = Path.cwd() / "agentic" / "knowledge-graph" / "graph.json"

if not graph_path.exists():
    return {
        "error": "Knowledge graph not found",
        "message": "Run /create first to generate agentic documentation and knowledge graph",
        "expected_location": str(graph_path)
    }
```

### Step 2: Load Graph into Memory

```python
import json
import networkx as nx
from networkx.readwrite import json_graph

# Load graph into NetworkX
with open(graph_path) as f:
    graph_data = json.load(f)

# Create in-memory graph
G = json_graph.node_link_graph(graph_data)

# Validate graph is not empty
if G.number_of_nodes() == 0:
    return {
        "error": "Empty knowledge graph",
        "message": "Graph exists but contains no nodes. Regenerate with /create"
    }
```

### Step 3: Parse Query and Execute

```python
# Query is executed entirely from in-memory graph
# NO file reads, NO database queries, NO external calls
# ONLY graph traversal and node/edge attribute access

query = "how does the installer work?"
query_type = classify_query(query)  # component-details
target = extract_target(query)      # "installer"

# Retrieve from graph
result = retrieve_from_graph(G, query_type, target)
```

### Step 4: Return Results

```python
return {
    "answer": result["answer"],
    "navigation_path": result["path"],  # e.g., ["AGENTS.md", "Components", "Installer"]
    "hops": len(result["path"]) - 1,
    "context_lines": result["context_size"],
    "related_links": result["related"]
}
```

## Auto-Cloning

If a GitHub URL is provided and not already cloned, the repository will be cloned to `/tmp/agentic-repos/<repo-name>/` before querying.

**Note**: The repository must have a knowledge graph at `<repo>/agentic/knowledge-graph/graph.json`. If not present, run `/create` first.

## Query Types

### 1. Component Discovery

**Questions**:
- "What components exist?"
- "List all components"
- "What are the main parts?"

**Response**:
- Read `AGENTS.md` (entry point)
- Extract component links from "Components" section
- Return list of 3-5 main components with brief descriptions

**Navigation**: 1 hop from AGENTS.md

### 2. Component Details

**Questions**:
- "How does {component} work?"
- "What does {component} do?"
- "Tell me about {component}"

**Response**:
- Navigate to component doc: `agentic/design-docs/components/{component}.md`
- Return: Purpose, Key Responsibilities, Dependencies
- Include links to related components

**Navigation**: 2 hops from AGENTS.md

### 3. Concept Lookup

**Questions**:
- "What is {concept}?"
- "Explain {concept}"
- "When to use {concept}?"

**Response**:
- Navigate to concept doc: `agentic/domain/concepts/{concept}.md`
- Return: Definition, Why it matters, When to use
- Include links to related concepts and components

**Navigation**: 2-3 hops from AGENTS.md

### 4. Architecture Overview

**Questions**:
- "Show me the architecture"
- "How is the system structured?"
- "What's the high-level design?"

**Response**:
- Read `agentic/DESIGN.md`
- Return: Design philosophy, core beliefs, architecture principles
- Include links to key components

**Navigation**: 2 hops from AGENTS.md

### 5. Development Guidance

**Questions**:
- "How do I contribute?"
- "How to set up development environment?"
- "How to run tests?"

**Response**:
- Read appropriate file: `agentic/DEVELOPMENT.md`, `agentic/TESTING.md`
- Return relevant sections
- Include related links

**Navigation**: 2 hops from AGENTS.md

### 6. Relationship Queries

**Questions**:
- "What uses {component}?"
- "What depends on {component}?"
- "How are {A} and {B} related?"

**Response**:
- Read component docs for both items
- Extract dependency information
- Show relationship chain
- Include architectural context

**Navigation**: 2-3 hops from AGENTS.md

### Progressive Disclosure

**Level 1**: Entry point (AGENTS.md)
- High-level overview
- List of components
- Quick navigation

**Level 2**: Component/Concept docs
- Detailed information
- Relationships
- When to use

**Level 3**: Deep details
- Implementation notes
- Code references
- Advanced patterns

**Key Principle**: Load minimal context, provide links to go deeper

## Implementation

When this skill is invoked:

1. **Parse Query**
   - Identify query type (component, concept, architecture, etc.)
   - Extract keywords (component names, concepts)
   - Determine required documentation files

2. **Retrieve from Knowledge Graph** (NEW)
   - Uses `retrieve-from-graph` skill internally
   - Efficient graph-based retrieval with relevance scoring
   - Automatic navigation path optimization
   - Respects ≤3 hop constraint and 500 line context budget

3. **Navigate Documentation** (Fallback)
   - Start from AGENTS.md (always)
   - Follow links to relevant docs (≤3 hops)
   - Load only necessary content
   - Used when knowledge graph not available

4. **Extract Information**
   - Read relevant sections from retrieved nodes
   - Filter to what's needed for query
   - Maintain context budget (max 500 lines)

5. **Format Response**
   - Provide concise answer
   - Include relevant details
   - Add navigation links
   - Suggest next steps

6. **Log Query**
   - Log all queries (for analytics)
   - Track which docs accessed
   - Monitor navigation patterns and hop counts

## Context Budget

**Maximum Context per Response**: 500 lines

| Document Type | Budget | Rationale |
|---------------|--------|-----------|
| AGENTS.md | 150 lines | Entry point only |
| Component doc | 100 lines | Single component |
| Concept doc | 75 lines | Single concept |
| Supporting doc | 100 lines | DESIGN, DEVELOPMENT, etc. |
| Code snippet | 50 lines | Illustrative only |

**Total**: Stay within 500 line budget per query

## Reconciliation Pattern

**Definition**: A control loop pattern where a controller continuously compares desired state (spec) with actual state (status) and takes action to reconcile differences.

**Why It Matters**:
- Core pattern in Kubernetes operators
- Enables self-healing and declarative configuration
- Used throughout OpenShift controllers

**When to Use**:
- Managing cluster resources
- Implementing operators
- Maintaining desired state over time

**How It Works**:
1. Watch for changes to resources
2. Compare spec (desired) vs status (actual)
3. Calculate diff
4. Apply changes to reach desired state
5. Update status
6. Repeat

**Related Concepts**:
- [Controller Pattern](./controller-pattern.md)
- [Operator Pattern](./operator-pattern.md)

**Used By**:
- Machine API Operator
- Cluster Version Operator
- All Kubernetes controllers

```

## Error Handling

**Documentation Not Found**:
- Error: "No agentic documentation found"
- Action: Suggest running `/create` first

**Component/Concept Not Found**:
- Error: "Component '{name}' not found in documentation"
- Action: List available components/concepts

**Ambiguous Query**:
- Warning: "Multiple matches found for '{term}'"
- Action: Ask user to clarify (component vs concept?)

**Broken Links**:
- Warning: "Referenced document not found: {path}"
- Action: Use available information, note issue
